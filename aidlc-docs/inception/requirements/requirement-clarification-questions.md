## Question 2 深掘り: プロジェクト情報の保存場所

各選択肢のメリット・デメリットを比較します：

| 方式                         | メリット                       | デメリット                       |
| ---------------------------- | ------------------------------ | -------------------------------- |
| A) ローカルJSON手動管理      | シンプル、ポータブル           | ユーザー負担大、同期不可         |
| B) VSCode/Kiroグローバル設定 | IDE統合、設定同期対応          | 他ツールから参照困難             |
| C) 各.kiro/から一覧化        | プロジェクト側で完結、分散管理 | スキャン必要、パフォーマンス懸念 |
| D) ~/.kiro-hub/一元管理      | 高速アクセス、専用管理         | 追加ディレクトリ、手動登録も必要 |

**推奨**: Q5でD（手動追加 + 自動検出）を選択されているので、**D) ~/.kiro-hub/ 一元管理**がベストフィットです。理由：
- 手動追加したプロジェクトの永続化に最適
- 自動検出結果のキャッシュとしても機能
- サイドバーツリービューへの高速ロードが可能
- プロジェクトメタデータ（カテゴリ、タグ、説明）の保存先として自然

この推奨を踏まえて選択してください：

A) D案を採用 — ~/.kiro-hub/ に一元管理（推奨）
B) B + D のハイブリッド — 基本はグローバル設定、メタデータは~/.kiro-hub/
C) C + D のハイブリッド — .kiro/から自動検出しつつ、登録情報は~/.kiro-hub/にキャッシュ
X) Other (please describe after [Answer]: tag below)

[Answer]: C

## Question 10 推奨: Property-Based Testing

このプロジェクト（Kiro拡張機能）の特性を考慮した推奨：

**プロジェクト特性**:
- VSCode/Kiro拡張機能（UI中心 + ファイルI/O）
- 主な処理: プロジェクト一覧管理、設定ファイルの読み書き、ツリービュー描画
- ビジネスロジック: プロジェクト検出ロジック、フィルタリング/ソート
- データ変換: JSON設定の読み書き、パス解決

**推奨: C) スキップ**

理由：
- 拡張機能は主にVSCode APIとの統合が中心で、PBTが効果的な純粋な計算ロジックが限定的
- テストはVSCode Extension Test Frameworkでの統合テストが主体になる
- プロジェクト検出やフィルタリングのロジックは通常のユニットテストで十分カバー可能

この推奨を踏まえて選択してください：

A) 推奨通り C（スキップ）を採用
B) B（部分的）を採用 — JSON設定のシリアライゼーションテストにのみ適用
X) Other (please describe after [Answer]: tag below)

[Answer]: A
